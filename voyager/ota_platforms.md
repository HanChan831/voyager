# 预订平台优先级说明

第1步用户已经选定了 `trip.booking_platform_preference`（默认"携程"），**这是第3/5/6步搜索交通/租车/住宿时的第一优先级**，搜索词里优先带上这个平台的名字。下面这份清单是这个偏好之外的补充参考——某个平台覆盖不到的时候按什么顺序补充搜，以及出境游场景的处理方式。

## 用户偏好平台查不到/覆盖不全时，按这个顺序补充

国内出行 / 目的地是国内城市：

1. **携程**（机票/火车票/酒店/门票综合性最强）
2. **飞猪**（机票/酒店，阿里系，常有专属优惠）
3. **去哪儿**（机票比价，年轻用户多）
4. **同程**（火车票/酒店，下沉市场覆盖较好）
5. 美团/大众点评（适合餐饮、本地玩乐类的预订和评价参考，不只是住宿交通）

如果用户偏好的平台本身就在这个列表里（比如选了"飞猪"），直接以飞猪为第一优先级，上面 1-5 的顺序仅供它查不到时往下试。

## 出境游 / 目的地是境外城市

用户偏好的国内平台（携程、飞猪这类）大多也覆盖国际机票酒店，可以继续作为第一优先级；如果某个目的地国内平台信息明显不全，再补充搜索国际平台作为参考：Booking.com（酒店）、Agoda（酒店，东南亚覆盖好）、Airbnb（民宿）、Skyscanner（机票比价）。

## 在最终方案里怎么呈现

1. 给用户呈现选项时，标注清楚是通过哪个平台查到的价格，例如："携程上查到 G1234 次二等座约 ¥553"。
2. 确认某个选项后，把平台名称写入 `trip_data.json` 对应 item 的 `booking_platform` 字段（如 `"携程"` `"飞猪"`），如果有信心这个搜索结果链接稳定可用，也可以填 `booking_url`；没把握的话留空，脚本会自动退回到该平台的官网首页链接。
3. **不要凭空构造平台的搜索结果深度链接**（比如试图拼出携程机票查询的完整 URL 参数）——这类链接通常依赖城市代码、动态 token 等不公开或易变的参数，构造错了比不给链接更糟（用户点开是错误页面）。除非你已经联网核实过某个平台确实存在公开稳定的查询 URL 格式，否则一律使用"平台首页链接 + 提示用户搜索关键词"这种保守但可靠的方式。
4. 在最终 HTML 文档里，这些链接会显示为每个交通/住宿项目旁边的"🎫 平台名"按钮（由 `itinerary_html_builder.py` 自动生成），点开后用户能直接跳转到对应平台开始查询/下单。
