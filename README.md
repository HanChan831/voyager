## 💬 前言

每个人的朋友圈里，是不是都有这样一个存在——

你还沉浸在"要不今年去趟日本吧"的念想里，他已经把往返航班、酒店比价、每日行程、备用方案全部排好了。当你在小红书上漫无目的地刷"东京必吃"的时候，他甩过来一份逐日时间线：几点从哪出发、坐什么线路、走几分钟到下一个点、午饭在哪吃人均多少、下午的景点要不要提前预约。当你终于开始犹豫"签证怎么办"的时候，他早就查好了：你的护照有效期够不够、材料清单列好了、建议从第几周开始办、连护照照片的尺寸要求都标出来了。

你忍不住想：**要是这个人能随时出现在我需要的时候就好了。**

Voyager 就是这位朋友。

它不是那种你问一句它回一段百科介绍、说完就消失的聊天机器人。它是一套**拒绝含糊、拒绝跳过、拒绝"差不多得了"**的旅行规划流水线。只要你给一个目的地，它就一步步把机票、住宿、餐饮、景点、签证、天气、预算、行前清单全部问清楚、搜清楚、算清楚——最后交到你手上的不是一段聊天记录，而是一个任何设备打开就能照着走的互动路书。

## 🧠 它能做什么

大多数旅行工具默认你已经知道自己想要什么。Voyager不预设你"应该知道"。它用一层层递进的选择题，帮你发现自己想要什么。

大多数网上攻略经不起四件事的推敲：

| 拷问 | 网上常见攻略 | Voyager |
|---|---|---|
| **花多少钱？** | "人均大概几千块吧" | 逐日逐项核算，按天、按类别汇总，含人均、总计、10-15% 应急缓冲金 |
| **时间排不排得开？** | 同一天塞了五个景点，通勤时间完全没算 | 自动化时间冲突检测，超 12 小时或空档过大会报警 |
| **能不能直接照着走？** | 复制粘贴了大段文字，到了当地找不着路 | 每个地点带导航链接（高德/百度/Google/Apple），点一下开始导航 |
| **入境手续搞清楚了吗？** | "好像免签吧" | 强制联网搜索，至少两个信源交叉验证，给出完整材料清单和办理时间倒推 |

### 核心能力一览

| 板块 | 做什么 | 产出 |
|---|---|---|
| 🌍 签证与入境 | 联网搜索目的地签证要求，至少两个信源交叉验证 | `visa_notes.md` —— 完整材料清单 + 办理时间倒推 |
| ✈️ 长途交通 | 搜索真实票价，默认推性价比最高方案，同时列出更快/更贵选项 | 含班次、价格、预订平台、枢纽配套服务 |
| 🚗 当地交通 | 公共交通估算 / 自驾租车全流程（驾照核查 + 交通规则 + 停车 + 保险） | `local_needs_notes.md` |
| 🏨 住宿 | 携程/飞猪等平台搜索，含评分、点评摘要、配套服务 | 依预算档位推荐 2-3 个选项 |
| 🍜 餐饮 | 小红书 + 点评双源搜索，预算区间或具体餐厅推荐二选一 | 人均每餐价 + 时间线嵌入 |
| 🎫 景点与逐日编排 | 按地理位置就近分配，兼顾风格偏好与节奏偏好，显式预留通勤和用餐时间 | 每小时时间线 + 门票/活动费用 |
| 🌤️ 天气 | 两周内查预报，超过两周查气候特征 | `weather_notes.md` + 穿衣/装备建议 |
| ✅ 行前清单 | 从签证/本地需求/通用菜单里摘取行动项，按紧迫度排序 | 可点击打勾的 HTML 清单 |
| 💰 预算核算 | Python 脚本自动算，不靠心算 | 按天 → 按类别 → 小计 → 总计 → 人均 |
| ⏱️ 时间核验 | 检测时间重叠、超时、空档 | 警告 + 修正建议 |
| 🗺️ 互动地图 | Leaflet + OSM，所有地点标记，可按天切换 | 点地图 marker 定位，多日路线 |
| 📸 地点图鉴 | 照片 + 评分 + 点评摘要 + 配套服务 | `places_gallery.html` |
| 📄 PDF 导出 | 浏览器原生打印，零依赖 | 页面顶部一键按钮 |

此外它有两个硬原则：

1. **不编造**——搜不到就是搜不到，评分留空也不填假数字，价格波动大就明确写"这是搜索时刻的参考价"。
2. **不跳过**——15 步必须走完，发现冲突必须修正后才能交付，不接受"差不多得了"。

## 🏛️ 设计哲学

### 1. 一条消息只问一个问题

不把语言、货币、预算、日期塞进同一条消息。宁可多来几轮，也不让用户不知道该先回哪个。

### 2. 能用字母回答的，不让用户打字

每个选择题都用 A/B/C/D 字母编号。回一个字母就够了——在手机上尤其重要。

### 3. 能联网核实的就不凭印象

价格、汇率、签证政策、营业时间、天气——一律搜。搜不到就说搜不到，不估一个"看起来合理"的数字。

### 4. 脚本核算，不靠人脑

心算会出错、手抄会漏项。所有费用和时间都由 Python 脚本处理：

```
geocode_lookup.py       → 自动补全所有地点坐标（OSM/Nominatim）
photo_lookup.py         → 自动查找地点照片（Google/Bing Images）
budget_calculator.py    → 按天×类别核算预算
schedule_checker.py     → 检测时间冲突
itinerary_html_builder.py → 生成互动地图 + 时间轴 + 日历 + 图鉴
```

### 5. 交付的是工具，不是文档

最终产物是一个**自包含的 HTML 文件**（+ 可选的本地照片文件夹），任何设备的浏览器都能打开。不只是阅读——页面上的清单可以打勾、地图可以点击、导航链接一键跳到地图 App、日历事件一键加入系统日历。

### 6. 最后一公里：一致性检查

内容产完之后不是直接交付——先过 16 项交叉检查：时间有没有矛盾、金额有没有手误、签证时间线是否合理、驾照结论是否具体到用户实际驾照类型、用户提出的修改请求是否全部落实……一项没通过就回退修正，而不是"发现了但先这样吧"。

## 🔄 完整流程

```
 0. 初始化工作区（创建目录 + trip_data.json）
  ↓
 1. 收集基础信息（语言→货币→预订平台→出发地→目的地→
    日期→天数→人数→预算→节奏→风格→护照→导航App）
  ↓
 2. 签证与入境材料核查（强制联网，至少两个信源）
  ↓
 3. 长途交通（搜索票价，推荐+列出选择，用户确认）
  ↓
 4. 当地交通（公交/自驾/其他三选一）
  ↓
 5. 本地特殊需求核查（驾照→交通规则→租车→停车→保险）
  ↓
 6. 住宿（搜索+评分+推荐+确认）
  ↓
 7. 餐饮（预算区间 or 具体推荐，含小红书搜索）
  ↓
 8. 景点与逐日编排（按偏好选景点+排时间线+天气查询）
  ↓
 9. 行程确认与补充（两次确认：行程调整 + 个人随身事项 +
    当地行为规范与禁带物品研究）
  ↓
10. 行前准备清单汇总（签证行动项+本地需求行动项+通用项+
    天气建议+个人事项）
  ↓
11. 核算与生成（5 个脚本顺序运行，不可乱序、不可跳过）
  ↓
12. 生成最终网页（模板替换→填入所有脚本输出）
  ↓
13. 最终一致性检查（16 项交叉核对，发现问题→修正→重检）
  ↓
14. 交付与收尾（打开浏览器预览 + 总结交付 + 分享方案说明）
```

每一步的详细规则写在 [`SKILL.md`](./voyager/SKILL.md) 中，约 500 行。如果你要改这个 Skill 的行为，读那个文件。

### 渐进式加载

Claude Code 的 Skill 机制支持渐进式加载：`SKILL.md` 在触发时加载进上下文，而 `SKILL.md` 中引用的参考文件（`visa_checklist.md` 等）只在执行到对应步骤时按需读取。这样既保证了每个步骤都能拿到详细指南，又不把上下文一口气撑爆。

## 📦 安装

### 前置条件

- [Claude Code](https://claude.ai/code) 已安装并配置好 API 密钥
- Python 3.x（所有脚本仅依赖标准库，无需 `pip install` 任何东西）

### 安装方式一：Git 克隆（推荐）

```bash
# 克隆到 Claude Code 的 skills 目录
git clone https://github.com/YOUR_USERNAME/voyager.git ~/.claude/skills/voyager

# 或者在项目本地使用
git clone https://github.com/YOUR_USERNAME/voyager.git ./my-project/.claude/skills/voyager
```

1. 下载本仓库 ZIP 并解压
2. 将整个文件夹放到 `~/.claude/skills/` 或项目的 `.claude/skills/` 目录下
3. 确保 Claude Code 能识别 `voyager/SKILL.md`

### 验证安装

在 Claude Code 中输入：

```
帮我规划一次旅行
```

如果 Claude 开始按 15 步流程询问你"请选择沟通使用的语言 / Please choose the language"，说明安装成功。

你不需要手动输入 `/travel-planner`——只要你的话听起来像要规划旅行，Skill 就会自动触发。以下任何一句话都行：

## 💡 使用技巧

### 技巧 1：第一句话多说一点，后面就少问一点

Skill 会逐步收集信息，但如果你第一句话就把关键信息说清楚了，对应步骤会自动跳过：

| ❌ 效果差 | ✅ 效果好 |
|---|---|
| "帮我规划一次旅行" | "帮我规划一次去东京的旅行，12月15号出发，玩5天，2个人，预算舒适型，住宿想住新宿附近" |

你说的越多，跳过的步骤越多，对话越短。

### 技巧 2：预算定位影响的不只是"贵不贵"

- **经济优先**：所有环节默认推性价比最高的方案；住宿往青旅/连锁酒店靠；餐饮给预算区间而非具体餐厅推荐；长途交通优先高铁/大巴而非飞机。
- **舒适优先**：住宿、餐饮、体验项目可以上更好的档位，但长途交通默认**仍然推荐性价比最高的方案**——更贵的选择会列出来让你自己决定要不要加钱。
- **自定义预算**：直接给一个总预算数字，或分类预算（如"住宿每晚不超过 400"），Skill 会据此反推所有方案。

### 技巧 3：节奏偏好影响的不止"每天去几个景点"

- **紧凑**：每天尽量排满，景点之间用最短交通方式衔接，留白少。
- **休闲**：多留白、慢慢逛，景点之间预留更多缓冲时间，不为"打卡效率"牺牲体验。

这个选择会直接影响 `schedule_checker.py` 的阈值判定——同样的时间安排，紧凑模式下可能被视为"正常"，休闲模式下会被警告"太赶了"。

### 技巧 4：两个"偏好"不要随便选

第 1 步有两个容易被随手选过去、但实际上非常影响景点推荐质量的问题：

- **旅行风格偏好**：热门打卡（知名地标/网红景点/拍照出片优先）vs 休闲度假（温泉/咖啡馆/市集）vs 都要
- **景观偏好**：自然风光（山川湖海/公园绿地）vs 城市人文（博物馆/老街区/建筑）vs 都要

选了"自然风光"+"休闲度假"，就不会给你排一连串 CBD 商圈的网红打卡点。第 8 步编排景点时，这两个偏好直接决定景点池子的筛选方向。

### 技巧 5：签证这一步，即使你觉得"肯定免签"也别跳过

第 2 步**强制执行**。很多人规划行程时完全忘了这件事，订完机票才发现来不及办——或者更糟："以为免签到了被遣返"。Skill 会：

- 至少用两个信息源交叉验证
- 查护照剩余有效期（很多国家要求 ≥ 6 个月）
- 查容易被忽略的电子入境登记（ESTA/eTA/K-ETA 这类）
- 按出发日期倒推"建议从第 X 周开始办理"
- **政策可能随时调整，临近出发前需要用户自己再次核实**

### 技巧 6：第 9 步的第二次确认，往往能挖出最重要的东西

第 9 步有两问：一是问行程要不要调整，二是问个人随身物品要不要提醒。第一问很多人直接选 A（没问题），但第二问常常能挖出关键信息：

> "对了，我对花生过敏，提醒我在当地注意一下"  
> "帮我提醒带晕车药"  
> "我要带无人机，帮我查一下能不能入境"

这些信息不仅会进入行前准备清单，还会跟"当地禁带物品研究"交叉核对——比如你要带无人机，但目的地禁飞或需要注册，会在清单里明确提醒。

### 技巧 7：照片和地图不用你操心

你不用在规划过程中手动找图。第 11 步的 `photo_lookup.py` 和 `geocode_lookup.py` 会自动用英文名 + 当地语言名分别搜索坐标和照片。知名景点自动匹配，冷门地点人工补一张就行。

**但要注意**：如果照片被下载到了本地 `img/` 文件夹，分享时必须把 `trip_plan.html` 和 `img/` 一起打包发出去，否则对方打开看不到照片。

### 技巧 8：中途想改，不用重来

直接告诉 Claude "把预算改成舒适型"或"出发日期改到 8 月 15 号"，它会修改 `trip_data.json` 然后重新跑脚本、重新过一致性检查。不会让你从第 1 步重来，也不会手工去改输出文件里的数字。

### 技巧 9：最终网页就是全部。PDF？点一下按钮就行

Voyager 只产出一个 `trip_plan.html`（+ 可选的本地 `img/` 照片文件夹）。需要 PDF？打开页面，点顶部的"打印/导出 PDF"按钮，在系统打印对话框里选"另存为 PDF"——浏览器就是 PDF 引擎，不需要 Skill 单独生成一份 PDF。

### 技巧 10：国内 vs 出境，自动区分

Skill 会根据目的地自动切换行为：

| 行为 | 中国内地 | 境外 |
|---|---|---|
| 评分来源 | 携程/大众点评/美团/高德 | Google Maps/Tripadvisor/Booking/Agoda/Yelp |
| 导航 App 推荐 | 高德/百度都可用 | 建议 Google 或 Apple（高德/百度境外数据不全） |
| 签证步骤 | 跳过（国内免签） | 强制执行两步信源核查 |
| 货币兑换提醒 | 不提醒 | 提醒 |
| 入境禁带物品 | 简化为文明出行提醒 | 完整搜索 |
| SIM 卡/漫游提醒 | 不提醒 | 提醒 |

```
trip-plan-workspace/

├── img/                    ← 地点照片（有本地照片时必须和 HTML 一起分发）
├── visa_notes.md           ← 签证与入境材料说明

├── customs_notes.md        ← 当地行为规范与禁带物品

├── itinerary_blocks.html   ← 逐日时间轴 HTML 片段
└── places_gallery.html     ← 地点图鉴 HTML 片段
```

### `trip_plan.html` 包含的功能

- 🗺️ **互动地图**：Leaflet + OSM，所有地点标记，可按天切换，点击列表定位 marker
- 🧭 **逐段导航**：每个地点之间有"新窗口打开路线"按钮，跳转高德/百度/Google/Apple 地图
- 📅 **加入日历**：每个有时间的地点一键添加到系统日历
- 📸 **地点图鉴**：照片 + 评分（带来源徽标）+ 点评摘要 + 配套服务说明
- 💰 **预算明细**：按天 → 按类别汇总，含小计/总计/人均/应急缓冲金
- ✅ **可打勾清单**：行前准备清单，点击即可打勾，刷新不丢失
- 🖨️ **打印/导出 PDF**：页面顶部按钮，浏览器原生打印，零依赖
- 🌤️ **天气卡片**：出行期间天气概况与穿衣建议
- 📋 **标签页导航**：概览 / 行前准备 / 逐日行程 / 地图 / 图鉴 / 预算 / 当地须知 / 签证

## ⚠️ 适配场景与局限

### 最适合

- 多日、多城市的自由行行程规划
- 需要真实价格和预订参考的预算核算
- 出境游（签证/货币/驾照/通讯每个环节都需提前准备）
- 自驾游（驾照核查、交通规则差异、停车信息）
- 对"时间排不排得开"有严格要求的行程

### 不太适合

- **当天往返的周边游**：15 步流程的体量对这种场景偏重，用普通对话更高效
- **纯跟团游**：交通、住宿、景点都已被旅行社安排好，没必要走完整核算流程
- **"我就随便看看、还没有真的要去"**：Voyager 假定你是真要出行，每一步的搜索都基于真实预订平台和当前时刻的数据。如果只是随便看看，建议直接用普通对话

### 需要用户自己二次确认的事项

Voyager 在以下环节会明确提示"请以官方/实际预订时为准"：

- 机票/酒店价格（搜索结果有滞后，波动大）
- 签证政策（可能随时调整，临近出发前需再次核实）
- 天气预报（超过两周的只是气候特征参考，不是精确预报）
- 汇率（实时变动，预算里的外币换算是参考值）

## ❓ 常见问题

<details>
<summary><b>Q: 规划一次行程大概需要聊多少轮？</b></summary>

取决于行程复杂程度和你提前给了多少信息：

- **简单国内游**（如成都 3 天）：约 15-20 轮对话
- **中等出境游**（如东京 5 天）：约 20-25 轮对话
- **复杂多城出境游**（如东欧三国 10 天）：约 25-30 轮对话

如果你第一句话就把出发地、目的地、日期、天数、人数、预算说清楚了，可以跳过至少 5-6 轮。
</details>

<details>
<summary><b>Q: 一定需要用 Python 脚本吗？</b></summary>

预算核算、时间核验、坐标查询、地图生成都由 Python 脚本完成。好处是：

1. 不会出现心算错误或手抄遗漏
2. 所有脚本**只依赖 Python 标准库**，不需要 `pip install` 任何第三方包
3. 如果没有 Python 环境，Skill 仍然可以完成对话规划和推荐，但最终无法生成互动 HTML——你需要手动整理结果
</details>

<details>
<summary><b>Q: 生成的 trip_plan.html 安全吗？能发给别人吗？</b></summary>

安全。`trip_plan.html` 是纯前端页面——不包含后端逻辑、不发送数据到任何服务器。地图使用免费 OpenStreetMap 数据 + Esri 瓦片服务，导航链接就是普通 URL，日历按钮生成 `.ics` data URI。发给任何人都安全。分享注意事项见 [`sharing_guide.md`](./voyager/sharing_guide.md)。
</details>

<details>
<summary><b>Q: 评分/点评/照片搜不到怎么办？</b></summary>

留空。找不到就填 `null`，页面上对应位置不显示。Voyager 的设计原则是：**空着比编一个假的数字更诚实、更有用**。

不过大部分知名景点和酒店都能通过自动化脚本匹配到，概率在 80% 以上。
</details>

<details>
<summary><b>Q: 中途可以改主意吗？</b></summary>

可以。直接告诉 Claude 要改什么，它会修改 `trip_data.json` 然后重新跑脚本。不会让你从第 1 步重来。第 9 步也专门设计了一次"要不要调整"的确认机会。
</details>

<details>
<summary><b>Q: 支持多目的地行程吗？</b></summary>

支持。多城市/多目的地的行程会被拆成多个子行程，每个独立走 3-8 步（交通→住宿→餐饮→景点），最后在第 10-13 步合并成一份完整路书。
</details>

<details>
<summary><b>Q: 跟直接让 Claude 帮我规划旅行有什么区别？</b></summary>

不加载 Voyager 时，Claude 也会帮你规划旅行——但它没有：

- 结构化的 15 步流水线（容易漏掉签证/驾照/禁带物品这类重要但容易被忽略的环节）
- 强制联网搜索验证（可能凭训练数据给出过时的信息）
- 自动化脚本核算（预算和时间靠手动算，容易出错）
- 16 项一致性交叉检查（前后矛盾的内容不会被揪出来）
- 标准化的互动 HTML 输出（输出格式不稳定，每次可能不一样）

简单说：**裸 Claude 能给出不错的旅行建议；加载 Voyager 后，Claude 能交出一份可以直接照着走的精确路书。**
</details>

<details>
<summary><b>Q: 可以集成到其他 AI 工具吗？</b></summary>

Voyager 是为 Claude Code 的 Skill 机制编写的（`SKILL.md` 格式）。Claude.ai、Anthropic API 理论上也能加载，但：

- 文件系统操作（创建目录、运行 Python 脚本、打开浏览器）是 Claude Code 特有的能力
- 在 Claude.ai 网页版上，脚本需要手动运行，互动地图功能也会受限制

Cursor 和 Codex CLI 等工具有类似的"规则文件"机制，但格式和加载方式不同，不能直接用。
</details>

<details>
<summary><b>Q: 能不能只做某一步，不走完整流程？</b></summary>

如果你想只查签证或者只算预算，可以直接说：

> "帮我查一下持中国护照去日本的签证要求"  
> "帮我算一下去成都 3 天的预算大概多少"

Voyager 会识别出你只需要某个特定步骤，跳过其余部分。但如果你上来就说"帮我规划旅行"，那就默认走进完整 15 步流程。
</details>
